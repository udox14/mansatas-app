## Stable candidates

### MANSATAS-026-ACADEMIC-IMPORT-AUTHZ — low-privileged or unauthenticated callers can replace academic assignments and schedules

- Instance key: `academic-import-authz:lib/academic-import.ts:347`.
- Affected assigned locations: `lib/academic-import.ts:237-240,245-341,347-400`.
- Minimum supporting locations: `app/dashboard/akademik/actions.ts:129-143` exposes the active-year ASC import without any authentication/feature check; `app/dashboard/akademik/actions.ts:563-585` lets any authenticated user create/reuse a draft for a caller-selected existing year without checking the `akademik` feature; `app/dashboard/akademik/actions.ts:613-635,666-704` accepts a caller-selected session ID and saves/applies it without authentication, feature authorization, or `created_by` ownership.
- Source: directly invoked Next.js server-action arguments: attacker-controlled XML, `tahunAjaranId`, `sessionId`, and wizard rows. A normal authenticated staff member lacking academic administration can create a draft for any known year; the wholly unguarded ASC/apply actions are additionally plausible unauthenticated entrypoints under direct server-action invocation.
- Closest apparent control: `createAkademikInputSession` calls `requireUser`, but checks neither role/feature nor requested year authorization, and returns a pre-existing shared draft regardless of creator. The page-level `checkFeatureAccess` is UI/navigation control, not a server-action boundary. `applyAcademicDataset` trusts the passed database/year/dataset and performs no authorization of its own.
- Sink/broken control: unconditional deletion of every `penugasan_mengajar` row for the supplied year at 347, followed by replacement assignments and schedules at 368-400; global subject metadata can also be updated at 261-281.
- Impact: unauthorized bulk destruction/replacement of teaching assignments and schedules in the active or historical academic year, with downstream attendance, agenda, and grading integrity impact. The selected year is bound correctly in SQL, but is not authorized for the caller.
- CWE: CWE-862 (Missing Authorization), CWE-863 (Incorrect Authorization), CWE-639 (Authorization Bypass Through User-Controlled Key).
- Candidate-local validation: invoke `createAkademikInputSession(historicalYearId)` as an ordinary feature-denied staff account, save a minimal valid dataset, then invoke `applyAkademikInputSession(sessionId)` and verify rows for that year were replaced. Separately test direct invocation of `importJadwalASC` without a session. Use disposable data because the operation is destructive.
- Attack path / preliminary disposition: internet or low-privilege staff -> directly invoked server action -> attacker-selected/shared draft and year -> `applyAcademicDataset` -> year-wide delete/reinsert. No server-side feature, CRUD, ownership, or role control intervenes. **Reportable, high confidence; likely high severity because the blast radius is year-wide academic integrity.**

### MANSATAS-001-PARENT-DEFAULT-CREDENTIAL — custom parent passwords do not revoke predictable NISN/DOB credentials

- Instance key: `auth-default-credential:utils/auth/index.ts:445`.
- Source: unauthenticated numeric NISN and password submitted through the public parent sign-in route (`app/api/auth/[...all]/route.ts:51-60`).
- Closest apparent control: PBKDF2 credential verification at 440-441. On mismatch, lines 445-458 still accept either the student's normalized NISN or `DDMMYYYY` birth date and overwrite the custom hash; the unenrolled path explicitly accepts the same values at 459-468.
- Sink: creation of a reusable 30-day `parent_session` and cookie at 504-535.
- Impact: anyone who knows/guesses a student's NISN or DOB can take over that parent identity, access the linked student's parent-visible records, and silently replace the chosen password. Password change cannot close the fallback.
- CWE: CWE-521, CWE-287.
- Candidate-local validation: change a test parent's password, then sign in using NISN and separately DOB; confirm session issuance and that the stored hash is replaced.
- Attack path / preliminary disposition: public endpoint -> student lookup -> predictable fallback -> session issuance. **Reportable; high-confidence improper authentication, at least medium severity for a bounded student's sensitive records.** This is the same canonical root cause already recorded by review 001-005 and should be deduplicated during synthesis.

### MANSATAS-001-STAFF-TOKEN-JSON — HttpOnly staff session token is exposed to same-origin script

- Instance key: `session-secret-exposure:utils/auth/index.ts:391`.
- Source/control: a valid staff cookie is parsed at 368-377 and protected by expiry/banned checks at 381-387. Cookie construction marks the bearer `HttpOnly` at 59-62.
- Sink: `getSession` includes `row.token` in its returned session at 389-392; `app/api/auth/[...all]/route.ts:73-78` serializes that object to JSON readable by browser script.
- Impact: a same-origin XSS can exfiltrate a 30-day reusable bearer token and replay it outside the browser, defeating HttpOnly's containment benefit.
- CWE: CWE-200 / CWE-922.
- Candidate-local validation: authenticated GET `/api/auth/get-session` and compare `session.token` with the cookie bearer. A same-origin script execution primitive is required for the remote attack chain.
- Attack path / preliminary disposition: victim cookie -> same-origin JSON request -> raw token -> exfiltration/replay. **Reportable defense break, likely medium when chained with XSS.** Deduplicate with review 001-005.

### MANSATAS-001-PARENT-TOKEN-JSON — HttpOnly parent session token is exposed to same-origin script

- Instance key: `session-secret-exposure:utils/auth/index.ts:570`.
- Source/control: valid parent cookie lookup and expiry check at 543-559; `HttpOnly` cookie construction at 59-62.
- Sink: `getParentSession` includes `row.token` at 568-574; `app/api/auth/[...all]/route.ts:81-86` serializes the object.
- Impact: same-origin script can extract and replay a 30-day parent session, extending transient script execution into off-browser access to the linked student's records.
- CWE: CWE-200 / CWE-922.
- Candidate-local validation: authenticated parent GET `/api/auth/parent/get-session` and confirm the JSON token equals the cookie bearer.
- Attack path / preliminary disposition: victim parent cookie -> JSON endpoint -> reusable token -> exfiltration. **Reportable, likely medium when chained with XSS.** Deduplicate with review 001-005.
