## Plausible raw candidates

### MANSATAS-067-CLASS-PERIOD-MIX — current class dashboard mixes historical discipline, agenda, and assignment-backed attendance data

- Instance key: `academic-year-read-mix:components/dashboard/KelasBinaanDashboard.tsx:205`.
- Source and authorization facts: `/dashboard/kelas-binaan?kelas=...` is browser-controlled, but the page resolves authenticated roles, obtains an accessible-class list, and accepts the requested ID only if it is in that list (`app/dashboard/kelas-binaan/page.tsx` 39-59). Ordinary wali access is thus confined to their current `kelas.wali_kelas_id`; leadership/admin roles intentionally receive a broader list. This is **not** a cross-class IDOR in the observed call graph.
- Period-mixing sinks: `topPelanggaran` sums all `siswa_pelanggaran` for students whose current `siswa.kelas_id` matches the selected class, without `sp.tahun_ajaran_id = taAktif.id` (205-213). `agendaTerbaru` selects assignments for the class with no `pm.tahun_ajaran_id` predicate (214-222), so old journal material can become the displayed "latest" material. `getFinalAttendanceForClass` is called for the current 30-day view (202-204); supporting helper inspection shows active-year schedule rows at `lib/wali-kelas-attendance.ts` 289-297 but assignment-backed attendance/session rows at 298-337 are filtered by class/date only, so records from old assignments can enter the maps during a year transition.
- Class/student facts: all named/PII-bearing rows are still limited to students currently assigned to the authorized class (current roster is built by the helper from `siswa.kelas_id`; local queries at 249-264 also require that class). The bug does not establish disclosure of another class or family. It can expose historical journal text and can inflate discipline/attendance risk indicators for current students.
- Closest controls: active `taAktif` is available as a prop (105), but it is used only by the shared teaching-schedule card (429), not these class aggregates. The selected class itself is caller-authorized as described above.
- Impact: bounded confidentiality/integrity issue at academic-year boundaries: a current wali may see previous-period journal content and current decision/risk summaries can be contaminated by old assignment records. Whether discipline is deliberately cumulative is a product-policy question; the schema's explicit `siswa_pelanggaran.tahun_ajaran_id` makes a period-specific interpretation plausible.
- CWE: CWE-639 / CWE-863 (period-scoped object authorization/integrity), with low confidentiality sensitivity for agenda text.
- Candidate-local validation: create year A and year B assignments for the same persistent class ID, activate B, seed a recent A agenda plus A attendance/session rows within the 30-day range and an A discipline event for a current student, then load the B class dashboard. Confirm old agenda/material and counts appear. Ask product owners whether discipline is cumulative; if yes, remove that leg but retain agenda/attendance mixing.
- Attack-path facts / preliminary disposition: authenticated current wali -> correctly authorized current class -> year-unscoped joins -> historical content/current dashboard calculations. **Preliminary: plausible low-to-medium integrity/confidentiality candidate; report only after the year-transition reproduction and policy check.**

### MANSATAS-070-AGENDA-PERIOD-MIX — private attendance card credits out-of-scope agenda rows against an active-year denominator

- Instance key: `academic-year-aggregation:components/dashboard/shared/KehadiranPribadiCard.tsx:136`.
- Source: persisted `agenda_guru` rows for the authenticated user during the current month/cutoff window. The numerator query filters only `guru_id`, date range, and effective date list (136-143); it does not join `penugasan_mengajar` or require the active `tahun_ajaran_id`.
- Closest control: the denominator correctly loads `jadwal_mengajar` for the active year and the same user (90-100), then applies effective dates, class deactivation, and KBM exceptions (117-134). That control does not constrain the independent agenda numerator.
- Sink and integrity effect: every out-of-scope status increments the current counts (145-155). `alfaBelumIsi = max(0, totalBlok - tercatat)` and `total = max(totalBlok, tercatat)` (157-165), so extra historical records both reduce computed absence and can increase displayed total/performance. A year activation within the same month naturally leaves prior-year rows in the queried window; malformed or directly created current-dated historical-assignment rows have the same effect.
- Authorization/object facts: current card usages pass the authenticated dashboard user's server-derived ID; no cross-user disclosure was established. The visible sink is the user's own performance card (175-190). A separate supporting-file inspection found that `submitAgenda` accepts a caller-chosen assignment and does not bind it to the caller/active year before persisting the assignment owner's `guru_id` (`app/dashboard/agenda/actions.ts` 224-241, 314-351); that is an upstream authorization concern owned outside this review, not merged here as a finding.
- Impact: current attendance/performance integrity can be inaccurate or gamed, but the assigned-file sink is private and does not itself prove that leadership, payroll, or formal evaluation consumes this card value.
- CWE: CWE-682, CWE-863.
- Candidate-local validation: activate year B mid-month after year A agenda rows exist for the same teacher and compare the card with a query joining `agenda_guru -> penugasan_mengajar` on B. Separately, in a reversible test environment, create a current-effective-date agenda tied to the teacher's historical assignment and confirm that `hadir/total` increases and computed `alfa` falls.
- Attack-path facts / preliminary disposition: historical/out-of-scope persisted agenda -> year-unscoped numerator -> active-year performance card. **Preliminary: low severity as observed; promote only if the same aggregation feeds formal staff evaluation or if the upstream action path is validated and consolidated by its owning review.**
