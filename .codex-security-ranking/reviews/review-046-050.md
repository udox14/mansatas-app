## Plausible raw candidates

### MANSATAS-046-BK-AUTHZ — unauthenticated/client-asserted BK authorization exposes and alters sensitive counseling records

- Instance key: `missing-action-authz:app/dashboard/bk/actions.ts:136`
- Affected locations: sensitive record read at 227-244; create at 246-268; arbitrary-ID edit/delete/session mutation at 271-311; bulk/list/report reads at 317-508; administrative assignment and synchronization at 148-168 and 514-566.
- Source: a direct server-action caller supplies the purported `guru_bk_id`, `tahun_ajaran_id`, `is_admin`, student/class/record IDs, and mutation payload. The action module never calls `getCurrentUser`, `getUserRoles`, or `checkFeatureAccess`.
- Closest apparent control: `tambahRekamanBK` checks that the **supplied** guru/year maps to the student's current class (252-258); report/list actions trust the **supplied** `is_admin` boolean (317-343, 377-396, 442-452). These checks do not bind either assertion to the session. Edit/delete/session functions do not even repeat the class/year relationship check (271-311).
- Sink/broken control: disclosure of counseling descriptions, interventions and follow-up notes (227-243, 399-434, 458-504), arbitrary counseling record alteration/deletion, and replacement of BK-class assignments.
- Academic-year facts: ordinary helpers accept any year ID. Only `sinkronKelasBinaanDariPenugasan` and `getInitialDataBK` derive the active year (517-525, 574-596). A caller can therefore use historical assignment rows or simply assert `is_admin=true`; current student membership is joined through mutable `siswa.kelas_id`, so the requested historical year is not tied to historical membership.
- Impact: unauthenticated or otherwise unauthorized disclosure and corruption of highly sensitive student behavioral/counseling records, with cross-class and cross-year reach. Administrative assignment mutation expands subsequent apparent access.
- CWE: CWE-306, CWE-862, CWE-639.
- Candidate-local validation: capture action identifiers from a production build, invoke `getRekamanSiswa(targetStudent, targetYear)` without a session, invoke `getLaporanBKSiswa(..., attackerChosenGuru, true)`, and attempt a reversible edit against a seeded record. Confirm no middleware-level action gate exists before reporting unauthenticated reachability. The code-local authorization failure is conclusive; only framework/deployment reachability remains to validate.
- Attack-path facts / preliminary disposition: browser-controlled action arguments -> no session/role/feature control -> privileged D1 binding -> sensitive rows/mutations. **Preliminary: reportable, high confidence; high severity because the same root spans bulk private records and administrative integrity.**

### MANSATAS-047-GUESTBOOK-AUTHZ — any authenticated account can enumerate and delete all visitor records

- Instance key: `role-authz:app/dashboard/buku-tamu/actions.ts:128`
- Affected locations: `getDataTamu` at 128-171 and `hapusEntriTamu` at 200-223; `getTamuHariIni` at 177-195 also exposes today's records.
- Source: any valid staff session and a caller-selected entry ID/filter. No role, feature, creator, or receptionist/admin relationship is checked after `getCurrentUser`.
- Closest apparent control: authentication only (132-133, 178-179, 201-202). The comments say monitoring admin/receptionist and “admin only” (126, 198), but neither claim is enforced. Deletion looks up and deletes solely by caller-supplied `id` (208, 221).
- Sink/broken control: bulk visitor name/organization/purpose/photo disclosure and deletion of both the database row and associated R2 photo.
- Impact: bounded but sensitive visitor privacy disclosure and destructive workflow interference by a low-privileged authenticated user.
- CWE: CWE-862, CWE-639.
- Candidate-local validation: using an account without the buku-tamu/admin feature, directly invoke `getDataTamu({page:1})`, then delete a seeded entry owned by another user and verify both DB and R2 removal. Proof gap: confirm whether product policy intentionally grants all authenticated staff guest-log access; comments strongly indicate it does not.
- Attack-path facts / preliminary disposition: low-privilege session -> direct action -> auth-only gate -> unrestricted query/delete. **Preliminary: medium, validation recommended.**

### MANSATAS-048-CKH-TEMPLATE-FEATURE-BYPASS — users lacking CKH feature access can directly create and mutate personal CKH templates

- Instance key: `feature-authz:app/dashboard/ckh-generator/actions.ts:864`
- Affected locations: template save/delete at 864-925 and note save/delete at 928-998.
- Source: any authenticated user, FormData/IDs, and their own user identity.
- Closest apparent control: core page/document functions call `requireCkhAccess` (338-344, 455-456, 499-500, and throughout 524-862). The four template actions call `getCurrentUser` and role lookup but never call `requireCkhAccess`.
- Sink/broken control: persistent creation/update/delete of personal `ckh_templates` and `ckh_template_notes`; non-global ownership checks do confine the caller to their own objects (881-895, 919-923, 941-963, 987-996).
- Impact: bypass of feature entitlement and unauthorized persistent data creation; blast radius is limited because ordinary users cannot modify global/other-user objects through this path.
- CWE: CWE-862.
- Candidate-local validation: remove `ckh-generator` feature entitlement from a test user, confirm page-data access rejects, then directly invoke each template action and verify mutations succeed. Establish whether all authenticated users are intentionally permitted template APIs independent of feature access.
- Attack-path facts / preliminary disposition: authenticated but feature-denied user -> template action omitting shared feature guard -> own-row write. **Preliminary: low/medium, retain for validation; not merged with CKH object ownership.**

### MANSATAS-048-CKH-ADMIN-PERSONAL-IDOR — CKH global-template administrators can overwrite/delete users’ private templates

- Instance key: `object-authz:app/dashboard/ckh-generator/actions.ts:881`
- Affected locations: existing template update/delete at 881-895 and 912-925; note update/reparent/create/delete at 941-963 and 980-998.
- Source: an authenticated `admin_tu` or `super_admin` supplies the ID of a template/note belonging to another user.
- Closest apparent control: `canManageGlobal` is derived server-side from roles (869-870, 917-918, 933-934, 985-986). But every ownership rejection is conditional on `!canManageGlobal` (884, 921, 943, 953, 994), so global-management authority also disables personal-object isolation. Save can additionally move another user's note into a selected template (945-958).
- Sink/broken control: update/delete of another user's personal CKH template and update/delete/reparenting of its notes.
- Impact: integrity/availability loss for individual staff performance-document presets. The caller is privileged, but the authority appears intended for global templates rather than private user data.
- CWE: CWE-639, CWE-863.
- Candidate-local validation: as `admin_tu`, pass an ordinary user's template/note ID and verify modification succeeds. Product-policy proof is required: if global CKH managers are explicitly authorized to administer every personal template, suppress this row.
- Attack-path facts / preliminary disposition: global-template manager -> caller-chosen personal object ID -> ownership condition bypassed -> mutation. **Preliminary: medium at most; policy validation required.**

### MANSATAS-050-KELAS-AUTHZ — class-management actions lack authentication and enable student moves plus role grants

- Instance key: `missing-action-authz:app/dashboard/kelas/actions.ts:53`
- Affected locations: class CRUD/import at 53-180; class/wali/KBM mutations at 186-429; student listing/assignment/mutation at 434-565; BK mapping reads/replacement at 572-612.
- Source: direct server-action arguments/FormData from an unauthenticated or unauthorized caller. The module has no auth/session/feature imports.
- Closest apparent control: object-existence and KM membership checks (85-90, 199-203) validate data consistency only. `syncWaliKelasRole` directly grants/revokes `wali_kelas` based on requested assignments (15-47). The “super_admin only” comment before BK assignment (567-590) is not executable authorization.
- Sink/broken control: persistent class creation/deletion, automatic `user_roles` changes, school-wide KBM state changes (323-360), student class assignment/swaps (446-565), and active/historical BK assignment replacement (590-608).
- Object-integrity facts: `prosesMutasi` does not verify that `siswaIdLama` currently belongs to `kelasIdLama`, that `siswaIdBarter` belongs to `kelasIdTujuan`, that target classes exist/share the expected grade, or that capacity is respected (505-532). Thus even a future coarse role gate would still need object validation.

- CWE: CWE-306, CWE-862, CWE-639, CWE-863.
- Candidate-local validation: capture deployed action identifiers and invoke a reversible `setStatusKbmKelas` or seeded student move without a session; verify `setWaliKelas` adds a `user_roles` row. Separately test inconsistent swap IDs. Confirm no deployment middleware rejects unauthenticated server actions.
- Attack-path facts / preliminary disposition: browser-controlled action -> no identity/role/feature gate -> privileged D1 writes and role sync. **Preliminary: reportable, high confidence; high severity due systemic integrity and privilege effects.**
