## Plausible raw candidates

### MANSATAS-071-HISTORICAL-TEACHER-SCOPE — an old teaching assignment grants access to the class's current student attendance data

- Instance key: `cross-period-authz:app/dashboard/rekap-absensi/actions.ts:140`
- Affected locations: `app/dashboard/rekap-absensi/actions.ts:122-150` (entrypoint scope/root control); `lib/wali-kelas-attendance.ts:260-337` (shared sensitive-read implementation); representative server-action sinks `app/dashboard/rekap-absensi/actions.ts:266-279,491-541,583-624,689-744`.
- Attacker-controlled source: an authenticated ordinary teacher can directly invoke the rekap server actions with a class/student identifier and requested dates. The teacher's durable user ID selects all rows in `penugasan_mengajar` at caller lines 140-144.
- Broken control: the teacher branch does not constrain `penugasan_mengajar.tahun_ajaran_id` to the active or explicitly selected academic year. Schema evidence confirms `penugasan_mengajar` is year-scoped (`migrations/schema.sql:156-163`), and multiple negative controls elsewhere use `pm.tahun_ajaran_id = ?` (for example `components/dashboard/shared/JadwalMengajarToday.tsx:20`). Every matching historical `kelas_id` is added to `allowedClassIds` at 146-150, and `canAccessClass` treats membership in that array as sufficient at 180-183.
- Sink/impact: after that check, `getFinalAttendanceForClass` returns the class's **current** active roster including names, NISN, gender/photo fields and detailed attendance, wali notes, permission reasons, reporter names, and lesson notes (`lib/wali-kelas-attendance.ts:268-337,439-515`). `getSiswaByKelas` also returns current names/NISN/NIS-local at caller lines 266-279. A teacher who taught class X only in a prior year can therefore enumerate and inspect current students in X, including students the teacher has never taught. This is a bounded but sensitive cross-academic-year confidentiality failure.
- Closest apparent control: authentication is present in each exported action and admins/wali/BK have intended branches. The teacher-specific relationship query is the incomplete control; no active-year join is applied. Downstream checks merely reuse its overbroad `allowedClassIds` and cannot repair it.
- Taxonomy: CWE-862 (Missing Authorization), CWE-639 (Authorization Bypass Through User-Controlled Key).
- Candidate-local validation: seed teacher T with a `penugasan_mengajar` row for class K in inactive year Y1, no assignment to K in active year Y2, and current Y2 students in K. As T, call `getSiswaByKelas(K)` and `getAbsensiSiswaKelasRentang(K, current dates)`; expected vulnerable result is success plus Y2 roster/attendance. Then add `tahun_ajaran_id = activeYearId` to the scope query as a negative control and confirm denial.
- Attack-path facts / preliminary disposition: authenticated teacher -> unfiltered historical assignment -> `allowedClassIds` -> attacker-selected current class/student -> current PII and attendance-note reads. No feature visibility or client-side navigation check intervenes in direct server-action invocation. **Preliminary: reportable, high confidence, likely medium severity under the supplied calibration.**

### MANSATAS-071-HISTORICAL-SNAPSHOT-MIX — historical attendance reports combine the current roster and active timetable with old dated records

- Instance key: `cross-period-snapshot:lib/wali-kelas-attendance.ts:268`
- Affected locations: `lib/wali-kelas-attendance.ts:268-296` (root control/current snapshot selection); `lib/wali-kelas-attendance.ts:298-337` (mixed-period record reads); `lib/wali-kelas-attendance.ts:392-515` (derived report); `app/dashboard/rekap-absensi/actions.ts:500-541,689-744` (browser-controlled range and report/print sinks).
- Attacker/operator-controlled source: authenticated reporting users choose arbitrary start/end dates (31-day-limited for the interactive range action, but the print action at 689-700 has no equivalent maximum). These dates can target an inactive academic year.
- Broken control: regardless of the requested dates, the helper selects students from mutable current state (`siswa.kelas_id`, `status = 'aktif'`) at 268-273 and selects the single currently active year's timetable/pattern at 274 and 290-296. It then reads `absensi_siswa`, wali decisions, permissions, and session rows for the requested dates without joining those rows to the period whose roster/timetable was selected at 298-337. The repository already preserves the correct year-keyed roster in `riwayat_kelas`, including immutable class snapshot fields (`migrations/schema.sql:189-199`; `lib/riwayat-kelas.ts:1-24`), but this helper never resolves the requested range to `tahun_ajaran_id` or joins `riwayat_kelas`.
- Sink/impact: historical JSON summaries and printable monitoring sheets can omit former class members, include current students who were not members then, calculate expected blocks using today's active timetable, and label missing historical rows as attendance-derived statuses. This can produce materially false official attendance output and may reveal old per-student notes whenever a current student has records matching the reused class ID/date selection. It is distinct from the authorization candidate: even an authorized admin/wali receives a cross-period data-integrity result.
- Closest apparent control: `getEffectiveDatesInRange` removes holidays and `kbm_nonaktif_mulai` truncates disabled classes, but neither selects the academic period, historical roster, or historical timetable. SQL binding prevents injection but does not repair temporal association.
- Taxonomy: CWE-682 (Incorrect Calculation), CWE-840 (Business Logic Errors); confidentiality aspect may additionally map to CWE-200 if cross-period note disclosure is reproduced.
- Candidate-local validation: create Y1 and Y2 with different timetables, move students between classes while retaining `riwayat_kelas` snapshots, and enter Y1 attendance. Request/print a Y1 range while Y2 is active. Compare returned roster, `dates`, `total_blok`, and statuses with Y1 snapshots and schedule. Confirm whether current students with prior same-class records receive old notes; that determines whether the finding is integrity-only or also confidentiality-impacting.
- Attack-path facts / preliminary disposition: authorized report caller -> selected historical dates -> current roster/current active timetable + historical dated rows -> returned/exported report. **Preliminary: validation recommended; strong static business-logic defect, security severity likely low-to-medium depending on whether generated output is relied on as an official record and whether cross-period notes are disclosed.**
