## Plausible raw candidates

### MANSATAS-058-PLOTTING-AUTHZ — unauthenticated plotting actions disclose student PII and alter school-wide class/graduation records

- Instance key: `missing-action-authz:app/dashboard/plotting/actions.ts:56`.
- Source: direct server-action arguments from an unauthenticated or unauthorized caller: student IDs, arbitrary class IDs, arbitrary source/target academic-year IDs, grade filters, and bulk payloads. There is no `getCurrentUser`, role lookup, or feature check anywhere in the module (imports 4-7).
- Closest apparent control: `resolvePlottingContext` only resolves caller-supplied year IDs against existing rows and silently falls back (65-82). Database existence and status predicates are data-consistency controls, not caller authorization. The kelulusan client supplies a `confirm()` dialog (79-88), but direct action invocation bypasses it.
- Sinks: student/NISN/sex/current and historical class reads (109-265, 585-648); persistent penjurusan drafts (271-373); arbitrary current `siswa.kelas_id` updates and `riwayat_kelas` writes (379-477); mass graduation clearing class IDs (536-554); and mass restoration (651-686).
- Object/year facts: `simpanPlottingMassal` accepts unrelated `siswa_id`/`kelas_id` pairs and does not verify target class grade/capacity or that students belong to the selected source year (379-477). `prosesKelulusanMassal` does not restrict supplied IDs to active grade-12 students at all (536-554), so any student can be marked graduated. Arbitrary existing source/target years can be selected for draft/history writes.
- Impact: broad disclosure of student PII and systemic corruption of active/historical academic placement and graduation state. Activity logging records the unauthorized operation but does not prevent it.
- CWE: CWE-306, CWE-862, CWE-639.
- Candidate-local validation: capture deployed action identifiers and invoke a read plus reversible draft change without a session; then in a seeded environment submit a non-grade-12 student to `prosesKelulusanMassal` and an inconsistent student/class/year tuple to `simpanPlottingMassal`. Confirm no deployment middleware rejects unauthenticated server actions.
- Attack-path facts / preliminary disposition: browser-controlled action payload -> no identity/role/feature gate -> privileged D1 reads and writes. **Preliminary: reportable, high confidence, high severity.**

### MANSATAS-056-DELEGATION-OBJECTS — any authenticated user can create delegation rows for assignments/classes they do not own

- Instance key: `object-authz:app/dashboard/penugasan/actions.ts:289`.
- Source: any authenticated user supplies `tanggal` and an array of `{penugasan_mengajar_id, kelas_id, tugas}` (289-296).
- Closest apparent control: the honest UI sources matching IDs from `getJadwalUntukDelegasi`, whose query correctly limits active-year/day assignments to `pm.guru_id = user.id` (189-253), but `kirimDelegasiTugas` does not repeat that control. It only validates nonempty text/reason and that some piket teacher exists for the supplied weekday (298-320).
- Broken object controls/sink: no query proves assignment ownership, active-year membership, a schedule on `tanggal`, or `pm.kelas_id = item.kelas_id` before arbitrary pairs are inserted (322-355). The caller can therefore manufacture an apparently legitimate incoming task for another teacher's assignment or a mismatched class, broadcast notifications, and create the prerequisite object used by delegated-attendance flows.
- Date/year facts: the date is arbitrary and converted only to a weekday. Piket schedules are not academic-year scoped here; assignment IDs are likewise not checked against the active year or date schedule.
- Impact: unauthorized workflow/assignment creation and misleading notifications; combined with the separate delegated-attendance identifier bug, a piket caller can use a manufactured delegation to reach attendance corruption.
- CWE: CWE-639, CWE-862.
- Candidate-local validation: as an ordinary authenticated account with no ownership of target assignment, submit a known foreign assignment and mismatched class ID, verify both delegation tables and piket notifications are created. Establish whether product policy intentionally allows all authenticated staff to delegate; even if yes, assignment ownership and pair consistency remain required.
- Attack-path facts / preliminary disposition: authenticated browser -> caller-selected assignment/class/date -> auth-only action -> delegation DB writes/notifications. **Preliminary: medium alone; strong chain with `MANSATAS-056-DELEGATED-ATTENDANCE-IDOR`.**

### MANSATAS-056-DELEGATED-ATTENDANCE-IDOR — a rostered piket teacher can read any class and overwrite attendance for an unrelated assignment

- Instance key: `object-binding:app/dashboard/penugasan/actions.ts:616`.
- Source: a valid piket/legacy recipient supplies `delegasiKelasId`, `penugasanMengajarId`, `kelasId`, date, slots, and student/status rows to `loadSiswaDelegasi` or `simpanAbsensiDelegasi` (590-596, 689-695).
- Closest apparent control: both actions verify that the supplied delegation-class row exists on the supplied date and that the caller is its named recipient or appears on the weekday's piket roster (615-625, 702-715). This authorizes access to **that dtk row**, but the query selects neither its stored `penugasan_mengajar_id` nor `kelas_id`, so it cannot bind the other arguments.
- Disclosure sink: `loadSiswaDelegasi` queries the caller-selected class roster/NISN, assignment attendance, permissions, exit status, and wali notes independently (627-680). A valid dtk token for any class/date therefore unlocks another class by swapping `kelasId` and `penugasanMengajarId`.
- Mutation sink: `simpanAbsensiDelegasi` deletes all attendance for the caller-selected assignment/date, inserts arbitrary supplied student IDs/statuses/slots, and marks a potentially different dtk complete (717-763). It does not validate statuses, student membership, assignment/class relationship, or scheduled slots. Notifications are also issued for the attacker-selected assignment (766-782).
- Date/year facts: authorization checks a caller-supplied date against the delegation and only uses its weekday for roster membership; it does not bind an active academic year. The independent assignment can be historical/current and need not be the delegated assignment.
- Impact: cross-class PII/attendance-note disclosure and potentially bulk attendance corruption by a lower-privileged piket teacher; arbitrary student IDs can be attributed to an unrelated assignment and existing rows can be erased.
- CWE: CWE-639, CWE-862, CWE-863.
- Candidate-local validation: obtain one legitimate dtk row as a rostered piket test account, call load with another class/assignment, then call save with a reversible unrelated assignment/date and confirm deletion/insertion occurs while the original dtk is marked complete. Test malformed status and out-of-class student IDs.
- Attack-path facts / preliminary disposition: authorized dtk token -> unbound browser-controlled class/assignment/student fields -> sensitive D1 reads/deletes/inserts. **Preliminary: reportable, high confidence, high severity.**

### MANSATAS-060-REKAP-HISTORICAL-TEACHER — historical teaching assignments grant access to current class attendance and PII

- Instance key: `academic-year-authz:app/dashboard/rekap-absensi/actions.ts:140`.
- Source: any authenticated teacher invoking recap/filter/class/student/print actions; class/student/date IDs are browser-controlled, but access is mediated by `getRekapScope`.
- Closest apparent control: admins are role-derived, BK classes are explicitly filtered to the active academic year, and wali classes come from a dedicated helper (122-139). In contrast, ordinary teaching classes are selected with `SELECT DISTINCT kelas_id FROM penugasan_mengajar WHERE guru_id = ?` and no `tahun_ajaran_id` or active schedule join (140-148).
- Sink: those historical class IDs authorize current `siswa.kelas_id` roster/NISN reads, detailed daily/range attendance notes, heatmaps, per-student reports, and print output through `getClassRows`/`canAccessClass` (153-183, 266-278, 491-666, 689-780).
- Year/object facts: class IDs are stable while student membership changes. Thus teaching class X in an old year adds X permanently to scope and reveals the students currently in X, rather than the historical students/year the relationship represented. BK handling at 127-135 demonstrates that active-year scoping is expected.
- Impact: bounded but sensitive current student PII and attendance disclosure to a teacher with only a historical relationship to that class.
- CWE: CWE-862, CWE-639.
- Candidate-local validation: seed a teacher assigned to class X only in a historical year, ensure different students currently occupy X, and directly call filter, daily matrix, per-student, and print actions. Confirm `getAccessibleWaliKelasClasses` does not intentionally broaden current access independently.
- Attack-path facts / preliminary disposition: authenticated historical teacher -> unscoped assignment lookup -> class allowlist -> current roster/attendance sinks. **Preliminary: reportable after relationship test; medium severity per threat-model calibration.**
