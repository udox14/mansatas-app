## Plausible raw candidates

### MANSATAS-040-PIKET-SUBMIT-OBJECT-AUTHZ — any authenticated user can forge a piket record for another user's schedule

- Instance key: `object-authz:app/dashboard/agenda/actions-piket.ts:184`
- Affected locations: `app/dashboard/agenda/actions-piket.ts:184-212` (direct server-action entry and missing ownership/root control), `app/dashboard/agenda/actions-piket.ts:216-261` (caller-controlled timing/status and insert sinks), and `app/dashboard/monitoring-agenda/actions.ts:572-600` (downstream monitoring consequence).
- Source: any authenticated caller can directly invoke the server action with attacker-chosen `jadwal_id`, `shift_id`, `tanggal`, `slot_mulai`, and `slot_selesai` FormData fields at 192-197, plus an image.
- Closest apparent control: authentication at 185-186 and effective-user selection at 188-190. The read/list action demonstrates the intended schedule authorization predicate: `j.user_id = userId` or a matching `guru_ppl_mapping` row at 129-144. The mutation never queries `jadwal_guru_piket`, never applies that predicate, never verifies that `shift_id` belongs to the schedule, and never derives the schedule day or shift slots server-side. Calendar effectiveness at 203-206 is not object authorization.
- Sink/broken control: the attacker-selected schedule and shift are persisted alongside the caller's `user_id` at 250-261. The database schema permits this combination: `agenda_piket` separately references `user_id`, `jadwal_id`, and `shift_id`, with only `UNIQUE(user_id, jadwal_id, tanggal)` (`migrations/patch_04_align_with_mts.sql:88-100`); no constraint requires the user or shift to match the referenced schedule.
- Exploitation detail: omitting both slot strings skips the time-window block at 216-231; empty `slot_mulai` then produces `NaN` in the status calculation at 238-241, whose comparison is false, yielding `TELAT`. Supplying attacker-chosen parseable slots can instead make the current time fall inside the fabricated window and choose `HADIR` versus `TELAT`. Arbitrary effective dates are accepted; the action does not require today or validate that the schedule's `hari` matches the date.
- Impact: integrity corruption of staff piket attendance and evidence records. In particular, monitoring joins `agenda_piket` to a schedule only by `ap.jadwal_id` and date, not by the scheduled user's ID (`app/dashboard/monitoring-agenda/actions.ts:586-600`), so the attacker's record can make another teacher's schedule display as filled/HADIR or TELAT. It can also attach attacker-uploaded evidence to that victim schedule and interfere with legitimate submission through the broad pre-check at 208-212.
- CWE: CWE-862 (Missing Authorization), CWE-639 (Authorization Bypass Through User-Controlled Key), CWE-20 (Improper Input Validation).
- Candidate-local validation: using a normal authenticated account with no matching `jadwal_guru_piket`/PPL mapping, invoke the action with a known victim `jadwal_id`, its or another valid `shift_id`, an effective date matching that schedule day, attacker-chosen slots, and a valid small image. Confirm an `agenda_piket` row is inserted with attacker `user_id` and victim `jadwal_id`, then confirm monitoring renders the victim schedule from that row. Repeat in a clean fixture with omitted slots to confirm time gating is bypassed. Proof gap: deployed Next.js action-origin/CSRF checks do not affect a same-origin authenticated attacker, but the exact action transport identifier must be obtained from the rendered client/build for an end-to-end HTTP PoC.
- Attack-path facts / preliminary disposition: authenticated low-privilege browser -> direct server action -> no role, feature, schedule-owner, PPL, day, or shift validation -> D1 insert -> privileged monitoring consumes the forged row as the victim schedule's attendance. **Preliminary: reportable, high confidence; medium severity because it corrupts a bounded attendance workflow without broader privilege gain.**

### MANSATAS-036-NILAI-PDF-HISTORICAL-ASSIGNMENT — a former teacher relationship remains an export grant for the class's current roster

- Instance key: `academic-year-authz:app/api/pdf/nilai-harian/route.ts:21`
- Affected locations: `app/api/pdf/nilai-harian/route.ts:16-28` (authenticated export entrypoint), `app/dashboard/nilai-harian/actions.ts:271-301` (root assignment check and roster query), and `app/api/pdf/nilai-harian/route.ts:30-58` (PDF disclosure sink).
- Source: authenticated caller-controlled `penugasanId` query parameter at route line 21; stale assignment IDs can come from the caller's former legitimate use, history, logs, bookmarks, or identifier disclosure.
- Closest apparent control: `getRekapNilai` verifies `pm.id = ? AND pm.guru_id = ?` at `actions.ts:283-290`, which blocks other teachers' assignments. It does not join/filter `tahun_ajaran.is_active`, despite `penugasan_mengajar.tahun_ajaran_id` being the period boundary (`migrations/schema.sql:156-162`). By contrast, the normal assignment dropdown selects the active year in `app/dashboard/nilai-harian/actions.ts:37-63`.
- Sink/broken control: after any historical assignment with at least one owned header passes the route's nonempty-header gate at 27-28, `getRekapNilai` loads every currently active student in `pm.kelas_id`, including name and NISN, at `actions.ts:292-301`, and the route renders/returns the PDF at 30-58. `kelas` is a persistent object without a year column (`migrations/schema.sql:77-86`), so an old assignment points at the same class object whose current active roster is queried.
- Impact: bulk disclosure of current student identities/roster and the former teacher's retained daily-grade material across an academic-year boundary. NISN is fetched into the action result even though the current PDF template displays student names and grades rather than NISN; the concrete PDF disclosure is current names plus grade export, not NISN on the rendered page.
- CWE: CWE-863 (Incorrect Authorization), CWE-639.
- Candidate-local validation: create or locate a teacher-owned `penugasan_mengajar` from a non-active year with at least one `nilai_harian_header`, ensure its `kelas_id` currently contains active students, request the route with that assignment ID, and verify a 200 PDF containing the current roster. Negative control: use another teacher's assignment and verify 404. Proof gap: production lifecycle may delete old headers or replace class IDs each year, although the schema and queries do not enforce either assumption; severity depends on actual retention/reuse.
- Attack-path facts / preliminary disposition: authenticated former teacher -> remembered historical assignment ID -> owner-only but period-unbounded assignment check -> current active roster lookup by persistent class ID -> bulk PDF response. **Preliminary: plausible and validation recommended; medium if class IDs are reused and old headers retained, otherwise suppress as unreachable.**
