## Plausible raw candidates

### MANSATAS-051-PPL-MAPPING-AUTHZ — any authenticated staff account can create delegated teaching authority

- Instance key: `delegated-authority:app/dashboard/kelola-ppl/actions.ts:189`.
- Affected locations: `app/dashboard/kelola-ppl/actions.ts:189-229` (direct server-action entrypoint and write sink); `app/dashboard/kelola-ppl/page.tsx:16-23` (page computes but fails to enforce feature access); supporting consumer `app/dashboard/agenda/actions.ts:129-150` (mapping grants agenda schedule access).
- Source: authenticated caller-controlled `guruPplId`, `guruUtamaId`, and each `jadwal_mengajar_id` / `jadwal_piket_id`.
- Closest apparent control: only `getCurrentUser()` at 195-196. There is no `kelola-ppl` feature/administrator check, no verification that the target is actually PPL, no verification that the principal teacher owns each schedule, and no current-year check. The schema has foreign keys but no relational constraint tying the four IDs together (`migrations/0012_remove_pu_dependency_from_guru_ppl_mapping.sql:8-14`).
- Sink/broken control: deletion of all existing pair mappings at 200-201 and batch insertion of arbitrary replacements at 208-223.
- Impact: a low-privileged staff account can alter or destroy PPL delegations. By choosing a PPL-controlled account and active schedule ID, the attacker grants that account access to another teacher's agenda block: the downstream agenda query authorizes solely when `jm.id` appears in a mapping for the effective user (`app/dashboard/agenda/actions.ts:140-150`). Other repository consumers also use this table for attendance, assignments, RPPM, and daily-grade assignment discovery, making the authority change cross-feature.
- CWE: CWE-862 (Missing Authorization), CWE-639 (Authorization Bypass Through User-Controlled Key), CWE-284 (Improper Access Control).
- Candidate-local validation: sign in as a normal staff account lacking `kelola-ppl`, invoke `simpanMappingPPL` with a real PPL ID and another teacher's active `jadwal_mengajar.id`, then act as/sign in as that PPL and observe the block returned by agenda. Also test a schedule that does not belong to `guruUtamaId`; local code predicts it is accepted. Proof gap: deployed Next.js action transport and availability of a controllable PPL account should be confirmed dynamically.
- Attack-path facts / preliminary disposition: authenticated low privilege -> directly callable action -> no role/feature/object check -> durable authorization-table mutation -> delegated consumers trust the row. **Preliminary: reportable high-confidence privilege escalation; high severity if a PPL account is attacker-controlled, otherwise at least medium integrity impact.**

### MANSATAS-051-PPL-ANON-READ — exported PPL actions disclose staff identities, schedules, and delegation relationships without authentication

- Instance key: `anonymous-staff-schedule-read:app/dashboard/kelola-ppl/actions.ts:58`.
- Affected locations: 58-68, 70-106, 111-124, 130-171, and 176-184.
- Source: unauthenticated direct action calls; `guruUtamaId`, `guruPplId` select staff records.

- Sink/broken control: returned names/roles, substitution relationships, subject/class/day/period schedules, and duty schedules.
- Impact: anonymous enumeration of staff deployment and school operating schedules; bounded privacy and physical/operational intelligence exposure.
- CWE: CWE-306 (Missing Authentication for Critical Function), CWE-862, CWE-200.
- Candidate-local validation: call each action without session and verify nonempty results; compare with a direct request to the page, which redirects to login. Proof gap: confirm production action IDs/endpoints are discoverable/replayable in the built deployment.
- Attack-path facts / preliminary disposition: internet client -> exported server action -> D1 staff/schedule queries -> serialized result, with no identity control. **Preliminary: reportable if action reachability is validated; likely medium because data is operationally sensitive but contains no student grades/PII.**

### MANSATAS-053-GRADE-CLASS-IDOR — a teacher can use one owned grade header to enumerate any active class roster and NISNs

- Instance key: `grade-class-idor:app/dashboard/nilai-harian/actions.ts:112`.
- Affected locations: `getNilaiDetail` at 112-135.
- Source: authenticated teacher supplies an owned `headerId` plus an unrelated caller-chosen `kelasId`.
- Closest apparent control: header creator ownership is correctly checked at 120-124. However, the lookup selects only `id`; it never joins `header.penugasan_id -> penugasan_mengajar.kelas_id`, so it cannot compare the caller's class to the header's class.
- Sink/broken control: query/return of active students' IDs, full names, NISNs, grades, and notes for arbitrary `kelasId` at 126-135. Grade/note rows are normally null unless a matching detail exists, but the roster and NISNs are still disclosed.

- Candidate-local validation: create/own a header for class A, call `getNilaiDetail(headerA, classB)`, and verify class B names/NISNs are returned. No deployment-specific control appears between source and sink.
- Attack-path facts / preliminary disposition: teacher session -> owned header passes creator check -> unrelated class ID is trusted -> student PII returned. **Preliminary: reportable medium object-level authorization bypass.**

### MANSATAS-053-GRADE-STUDENT-INJECTION — grade batch upsert accepts students outside the header's assigned class

- Attack-path facts / preliminary disposition: teacher -> owned header check -> attacker-selected student FK -> unconstrained upsert. **Preliminary: reportable medium academic-integrity violation; raise severity if downstream report cards consume these rows by student.**

### MANSATAS-053-GRADE-YEAR-SPILL — historical assignments remain usable for current-year grade operations and current rosters

- Impact: a teacher's past relationship silently grants ongoing access to current students in that class and permits cross-period academic-data mutation, violating the explicit academic-year trust boundary.

- Candidate-local validation: use an assignment owned by the teacher whose `tahun_ajaran_id` is not active. Confirm `buatSesiNilai` creates a header tagged with the active year, `getRekapNilai` returns current active students from that class, and `simpanKKM` updates it. Proof gap: availability/reuse of historical assignment/class IDs in production data.
- Attack-path facts / preliminary disposition: historical ownership -> year-agnostic lookup -> current roster/current-year write sink. **Preliminary: reportable medium cross-academic-year access and integrity defect.**
