## Plausible raw candidates

### MANSATAS-045-ANALYTIK-GRADE-WRITES-NO-AUTH — unauthenticated server actions can rewrite or erase school-wide grade history

- Instance key: `missing-authz:app/dashboard/analitik/actions.ts:61`.
- Affected locations: `app/dashboard/analitik/actions.ts:61-188` and `194-206`; minimum supporting callers `app/dashboard/analitik/components/analitik-client.tsx:14,301` and the feature's page/menu policy.
- Source: a direct server-action caller supplies `dataExcel` plus one allowlisted semester column, or supplies only an allowlisted column to reset. The action file never calls `getCurrentUser`, `requireUser`, `checkFeatureAccess`, or a CRUD permission helper.
- Closest apparent control: the visible `/dashboard/analitik` page requires a staff session but itself has no feature check (`app/dashboard/analitik/page.tsx:63-76`); the menu declares this feature for `super_admin`, `kepsek`, and `wakamad` (`config/menu.ts:359-365`). Neither UI/menu control executes at the server-action boundary. Next server-action references are shipped to the client that imports them, so obscurity of the generated action identifier is not authorization.
- Sink/broken control: import first loads every student's NISN mapping (65-82), maps attacker-provided rows to student IDs (87-119), and batches updates/upserts into `rekap_nilai_akademik` (131-175). Reset executes an unqualified `UPDATE rekap_nilai_akademik` for the selected semester (194-201), erasing that column for every row.
- Impact: an unauthenticated attacker who invokes the exposed action can corrupt selected grades for many or all students or erase an entire semester column school-wide. This directly compromises a core academic-integrity asset and can affect graduation/SNBP analytics.
- CWE: CWE-306 (Missing Authentication for Critical Function), CWE-862 (Missing Authorization).
- Candidate-local validation: from a browser without a session, replay the action reference emitted for `importNilaiDariExcel` with a known test NISN and one recognized subject column, then read back the test record; separately invoke `resetNilaiKolom('nilai_smt1')` only in a disposable database and confirm all rows become `{}`. Runtime validation must never target production because reset is destructive.
- Attack-path facts / preliminary disposition: unauthenticated request -> publicly referenced server action -> only column allowlist validation -> bulk D1 write/global reset. The allowlist prevents SQL injection but does not authorize the operation. **Preliminary: reportable, high confidence, high severity.**

### MANSATAS-041-AGENDA-FORGE — any staff account can create an agenda under another teacher and bypass assignment/year/schedule/time integrity

- Instance key: `object-authz:app/dashboard/agenda/actions.ts:232`.
- Affected locations: `app/dashboard/agenda/actions.ts:224-358`, especially the browser tuple at 232-239, client-time checks at 274-302, incomplete assignment lookup at 314-336, and insert at 338-351.
- Source: any authenticated staff caller supplies `penugasan_id`, `tanggal`, `jam_ke_mulai`, `jam_ke_selesai`, `slot_mulai`, and `slot_selesai`. The assignment ID can name another teacher's assignment or a historical assignment.
- Closest apparent control: `getJadwalGuruHariIni` demonstrates the intended relationship and period predicates: active `tahun_ajaran` (119-126), `jadwal_mengajar.tahun_ajaran_id = active` (146), and either assignment ownership or a PPL mapping (147). `submitAgenda` uses none of those predicates. Its assignment lookup selects only owner/class fields by `pm.id` (315-320); it does not select/check `pm.tahun_ajaran_id`, join a schedule row for the submitted day/block, or compare the caller/effective user to `pm.guru_id`/PPL mapping.
- Sink/broken control: after lookup, the action deliberately sets `guru_id` to the assignment owner's ID (335-340) and inserts the caller's material/date/hours/status (338-351). Thus the persisted record appears to belong to the victim teacher. The only audit field is `diubah_oleh`, which does not prevent the forged operational record.
- Exploit details: caller-chosen `slot_mulai`/`slot_selesai` drive the time gate (274-290), so a non-act-as user can submit a permissive range such as the current time inside `00:00`-`23:59`. The date is checked only for calendar effectiveness (243-247), not today, schedule day, or assignment year. `jam_ke` values are likewise trusted for exception/status calculation. A stale assignment ID therefore crosses the active-year boundary and an arbitrary current assignment crosses the teacher boundary.
- Impact: a low-privileged staff account can falsify another teacher's statutory/operational teaching agenda, including historical or future effective dates, and can occupy the unique `(penugasan_id,tanggal)` slot so the legitimate teacher is told it is already filled. Repetition over obtained assignment IDs can affect many teachers.
- CWE: CWE-639 (Authorization Bypass Through User-Controlled Key), CWE-862 (Missing Authorization), CWE-602 (Client-Side Enforcement of Server-Side Security).
- Candidate-local validation: authenticate as staff A, obtain an assignment belonging to teacher B (or an inactive-year assignment), submit that ID with an effective date and forged broad slot range, then verify `agenda_guru.guru_id` equals B while `diubah_oleh` equals A. Also verify no matching active-year `jadwal_mengajar` row is required.
- Attack-path facts / preliminary disposition: authenticated staff -> arbitrary/stale assignment and client schedule tuple -> no ownership/active-year/schedule predicate -> record stored as victim teacher. **Preliminary: reportable, high confidence, medium severity; raise to high if these agenda records drive formal compliance or payroll without review.**

### MANSATAS-043-NILAI-PAGE-NO-FEATURE — every staff role can browse all active students' grades and reach grade-management UI

- Instance key: `feature-authz:app/dashboard/akademik/nilai/page.tsx:11`.
- Affected locations: assigned page lines 10-20; minimum supporting `app/dashboard/akademik/nilai/components/nilai-page-tabs.tsx:9-25`; supporting grade reader/mutators in `app/dashboard/akademik/nilai/actions.ts:40-221,263-329`.
- Source: any valid staff session, including a role without the academic-management feature, navigates directly to `/dashboard/akademik/nilai`.
- Closest apparent control: only `getCurrentUser` is checked (10-12). The sibling academic center and staged-input pages both call `checkFeatureAccess(db, user.id, 'akademik')` before data access (`app/dashboard/akademik/page.tsx:83-91`; `input-bertahap/page.tsx:28-35`). This page omits that same control.
- Sink/broken control: it always renders `NilaiPageTabs` (14-20), whose rekap tab invokes a supporting action that returns every active student's ID, NISN, name, gender, class and six semester grade objects, while the import tab exposes validation/import/global-reset flows. Those supporting actions also lack independent authorization, so the page omission is not rescued below the UI.
- Impact: staff roles not intended to manage academics can disclose a school-wide set of student identifiers and academic records. The supporting direct actions separately permit integrity changes; this page candidate's clearest assigned-file impact is broad confidential-data exposure.
- CWE: CWE-862 (Missing Authorization), CWE-200 (Exposure of Sensitive Information).
- Candidate-local validation: sign in as a staff role for which `checkFeatureAccess(..., 'akademik')` is false, request `/dashboard/akademik/nilai`, and confirm a 200 page followed by the rekap action returning students outside any teaching/class relationship.
- Attack-path facts / preliminary disposition: any staff session -> no feature/role gate -> all-student rekap action -> NISN/class/multi-semester grades. **Preliminary: reportable, high confidence, high severity due to school-wide academic-record exposure across roles.**

### MANSATAS-041-AGENDA-READ-IDOR — agenda reads let callers select any teacher, and detail has no authentication

- Instance key: `object-authz:app/dashboard/agenda/actions.ts:75`.
- Affected locations: `getJadwalGuruHariIni` at 62-197, `getAgendaDetail` at 364-380, and `getRiwayatMateri` at 386-416.
- Source: an authenticated caller supplies `guruIdOverride` and optionally `dateOverride` to the schedule reader or history reader; any caller supplies `agendaId` to the detail reader.
- Closest apparent control: schedule/history check only that some staff user exists (71-72, 387-388). When an explicit override is supplied, they skip `getEffectiveUser` and assign it directly (75-81, 391-397). The normal page derives `effectiveUserId` through the super-admin-only act-as helper, but that page choreography is not an action-level control. `getAgendaDetail` does not even call an authentication helper.
- Sink/broken control: arbitrary teacher schedule reads expose subject, class, teacher, assignment ID, fill state and agenda ID (130-193); history returns the selected teacher's last 100 material entries and class/date metadata (400-416); detail returns `ag.*`, material, photo URL, status, teacher, class and editor identity (364-380).
- Period facts: the schedule query correctly binds to the active year (119-150), so it does not itself leak a stale-year schedule. History intentionally spans years but becomes unauthorized because the teacher selector is uncontrolled. `dateOverride` permits calendar-valid arbitrary dates but schedule rows still come from the active-year timetable.
- Impact: a staff account can inspect other teachers' lesson history and current timetable/agenda state, then chain the disclosed agenda IDs into full detail. An unauthenticated caller with a separately learned high-entropy agenda ID can also retrieve full detail. Material/photo sensitivity is deployment-dependent, keeping severity bounded.
- CWE: CWE-639 (Authorization Bypass Through User-Controlled Key), CWE-862 (Missing Authorization), CWE-200 (Exposure of Sensitive Information).
- Candidate-local validation: as a normal staff user who is not acting as teacher B, call schedule/history with B's user ID and confirm B's results. Take a returned `agenda_id`, invoke detail both with the same account and without a session, and confirm the full record is returned.
- Attack-path facts / preliminary disposition: authenticated staff -> arbitrary teacher ID -> lesson/schedule metadata and agenda ID -> unauthenticated detail action -> material/photo/audit metadata. **Preliminary: reportable, high confidence, medium severity.**

### MANSATAS-045-ANALYTIK-SETTINGS-NO-AUTH — unauthenticated callers can read and replace global graduation-ranking inputs

- Instance key: `missing-authz:app/dashboard/analitik/actions.ts:37`.
- Affected locations: `app/dashboard/analitik/actions.ts:15-54`.
- Source: a direct caller supplies arbitrary `mapel_snbp`, `mapel_span`, `bobot_rapor`, and `bobot_um` values to `simpanPengaturanAkademik`; the getter also accepts unauthenticated invocation.
- Closest apparent control: none in the actions. The intended menu restricts `analitik` to leadership roles, but neither the action nor its page uses `checkFeatureAccess`. Input is typed `any`, and there are no range/sum/shape checks before serialization/update.
- Sink/broken control: `dbUpdate` replaces the singleton global settings row (39-50), which the analytics page consumes for SNBP/SPAN calculations. The getter returns the entire settings row at 15-34.
- Impact: an unauthenticated attacker can manipulate subject selection and weighting used by graduation/eligibility analytics. This does not directly modify source grades, but it can silently distort school-wide ranking outputs until corrected.
- CWE: CWE-306 (Missing Authentication for Critical Function), CWE-862 (Missing Authorization), CWE-20 (Improper Input Validation).
- Candidate-local validation: without a session in a disposable database, invoke the save action with conspicuous weights/subject arrays, call the getter, and confirm the singleton changed. Restore the row afterward.
- Attack-path facts / preliminary disposition: unauthenticated request -> direct action -> unvalidated global settings update -> altered ranking analytics. **Preliminary: reportable, high confidence, medium severity.**
