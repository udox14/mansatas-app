## Plausible raw candidates

### MANSATAS-035-EKSKUL-PDF-IDOR — a pembina can export another extracurricular's student attendance and identities

- Instance key: `object-authz:app/api/pdf/ekskul-kehadiran/route.ts:28`.
- Affected locations: `app/api/pdf/ekskul-kehadiran/route.ts:17-33` (entrypoint and broken root control); `app/dashboard/ekstrakurikuler/master/actions.ts:256-314` (data read); `app/api/pdf/ekskul-kehadiran/route.ts:35-59` (PDF disclosure sink).
- Source: authenticated caller-controlled `ekskulId` and `bulan` query parameters at route lines 27-30. A normal pembina with the broad `ekstrakurikuler` feature passes lines 23-25.
- Closest apparent control: the route verifies only broad feature membership. The normal object-level policy is explicit in supporting `app/dashboard/ekstrakurikuler/actions.ts:87-93`: membership in `ekstrakurikuler_pembina` for the selected `ekskulId`, or master/admin access. The normal list also restricts a non-admin to joined pembina rows at supporting lines 104-125. Neither control is called here. `getLaporanKehadiran` itself begins directly with `getDB()` and has no session/feature/object guard (master actions 256-261).
- Sink/broken control: the helper reads every active member's name, NISN, class, and per-meeting attendance for the selected object (master actions 265-310); the route renders and returns those records as a PDF (35-59).
- Impact: a pembina can enumerate extracurricular IDs and obtain bounded but sensitive student identity and attendance records for groups they do not supervise. Arbitrary historical `bulan` values are accepted, and the meeting query has no academic-year constraint, so historical meeting data can also be combined with the currently active member list.
- CWE: CWE-639 (Authorization Bypass Through User-Controlled Key), CWE-862 (Missing Authorization), CWE-200 (Exposure of Sensitive Information).
- Candidate-local validation: authenticate as a user who has `ekstrakurikuler` but not `ekstrakurikuler-master`, choose an `ekskulId` absent from that user's `ekstrakurikuler_pembina` rows, and request a month containing meetings; expect HTTP 200 and names/classes/attendance in the PDF. The repository code closes the authorization proof; only representative seeded IDs/months are needed for runtime reproduction.
- Attack-path facts / preliminary disposition: authenticated pembina -> broad feature gate -> arbitrary object ID -> unguarded D1 reads -> private PDF. The repository's adjacent `canManage` function is direct counterfactual evidence of the intended ownership boundary. **Preliminary: reportable, high confidence, medium severity.**

### MANSATAS-034-CKH-CLIENT-IDENTITY — any staff session can forge the identity and signers in CKH PDFs

- Instance key: `document-integrity:app/api/pdf/ckh/route.ts:38`.
- Affected locations: `app/api/pdf/ckh/route.ts:21-49` (entrypoint/root control and client data acceptance); `lib/pdf/ckh-template.ts:61-154` (supporting representation/signature rendering); route lines 53-76 (browser/PDF sink).
- Source: any authenticated staff caller can submit arbitrary JSON values for `rows`, `user`, `kepsek`, `kepalaTu`, `userRoles`, `signatureSettings`, `year`, and `month` (25-48). The intended client confirms these values are serialized from browser state at supporting `app/dashboard/ckh-generator/components/ckh-generator-client.tsx:390-398`, so they are not trusted server-only objects.
- Closest apparent control: only a valid staff session is required (21-23) and only nonempty `rows` is structurally checked (31-33). The intended page requires `ckh-generator` feature access at supporting `app/dashboard/ckh-generator/page.tsx:32-37`, and its action loads the session user's profile by bound `userId` at supporting `app/dashboard/ckh-generator/actions.ts:109-119`; neither server-side control protects this direct route.
- Sink/broken control: the template prints the submitted person's name, NIP, rank, position, selected approving signer, and optional submitted signature URL into a branded CKH document, after which Puppeteer returns the PDF (route 53-76).
- Impact: a staff account without CKH permission can generate convincing school performance documents under another employee's or approver's identity, undermining document integrity. This route does not persist or finalize a CKH record, which limits blast radius, but the output itself is immediately usable outside the application.
- CWE: CWE-862 (Missing Authorization), CWE-345 (Insufficient Verification of Data Authenticity), CWE-602 (Client-Side Enforcement of Server-Side Security).
- Candidate-local validation: authenticate as a staff user lacking `ckh-generator`, POST one row plus invented `user`/signer/NIP values and a disabled signature, and confirm a 200 PDF contains those identities. Then repeat using another known signature URL if validating signature misuse. Proof gap: no cryptographic seal is visible in this slice, so external recipients may or may not treat the PDF as authoritative.
- Attack-path facts / preliminary disposition: any staff session -> direct route bypasses page feature gate -> caller supplies represented identities -> branded PDF. **Preliminary: reportable integrity candidate, likely medium; calibrate lower if CKH PDFs have no authoritative workflow value until separately finalized.**

### MANSATAS-033-PARENT-LOGOUT-NOT-REVOKED — parent logout clears the browser but leaves the 30-day server session valid

- Instance key: `session-invalidation:app/api/logout/route.ts:21`.
- Affected locations: `app/api/logout/route.ts:7-28` (logout flow/root control); `utils/auth/index.ts:504-515` (supporting parent session creation/30-day cookie); `utils/auth/index.ts:542-575` (supporting parent-token lookup).
- Source: a parent presents the shared `session_token` cookie to the logout endpoint; independently, an attacker may already possess a copied parent bearer token.
- Closest apparent control: the route extracts the token, but only executes `DELETE FROM session WHERE token = ?` at 21. Parent tokens live in `parent_session` (`utils/auth/index.ts:509-512`), and the concrete parent lookup continues accepting a matching unexpired row at supporting lines 543-575. Clearing cookies at route lines 25-28 changes only this browser's state.
- Sink/broken control: replay of the supposedly logged-out token continues to authenticate via `getParentSession`, reaching parent-private records until expiry (session expiry is 30 days in `utils/auth/index.ts:45-62`).
- Impact: logout does not terminate a compromised parent session; a stolen token remains usable to access the linked student's parent-visible private data. The same omission exists in the normal dashboard logout action (`app/dashboard/actions.ts:9-31`), strengthening reachability beyond this legacy/direct route.
- CWE: CWE-613 (Insufficient Session Expiration).
- Candidate-local validation: log in as a parent, copy the token, call `/api/logout`, then replay the copied token against a parent-authenticated endpoint; expect authentication to continue because the `parent_session` row remains. Proof gap: select a safe parent endpoint for runtime confirmation.
- Attack-path facts / preliminary disposition: copied parent token -> victim logs out -> only staff table deletion and local cookie clearing -> token replay -> parent lookup succeeds. **Preliminary: reportable, high confidence, medium severity because exploitation requires prior token compromise but defeats an explicit revocation boundary.**

### MANSATAS-031-FCM-DELETE-NO-AUTH — anyone holding a registration token can remove another user's push subscription

- Instance key: `subscription-authz:app/api/fcm/register/route.ts:56`.
- Affected locations: `app/api/fcm/register/route.ts:56-62`.
- Source: unauthenticated caller-controlled JSON `token` at 58-59.
- Closest apparent control: none. Unlike POST, which resolves staff/parent identity at lines 8-23 and derives ownership from that session, DELETE never calls either session helper and does not compare `owner_type`, `user_id`, or `siswa_id`.
- Sink/broken control: unconditional `DELETE FROM fcm_tokens WHERE token = ?` at 60-61.
- Impact: targeted loss of security, attendance, schedule, or operational push notifications for a staff member or parent. Registration tokens are normally high-entropy bearer-like values, so an attacker must first obtain a victim token; the route does not create an enumeration oracle because it reports success for absent tokens too.
- CWE: CWE-862 (Missing Authorization), CWE-639 (Authorization Bypass Through User-Controlled Key).
- Candidate-local validation: register a token for user A, submit the same token to unauthenticated DELETE, then verify the row is absent and notifications no longer target it. Proof gap: establish how readily tokens can leak from the Android/web client before final severity calibration.
- Attack-path facts / preliminary disposition: public JSON request -> arbitrary token predicate -> deletion. **Preliminary: valid but bounded availability/integrity issue; low severity absent a separate token-disclosure path.**

### MANSATAS-033-LOGOUT-CSRF — cross-site top-level navigation can force logout

- Instance key: `csrf:app/api/logout/route.ts:7`.
- Affected locations: `app/api/logout/route.ts:7-32`.
- Source: an attacker causes a victim browser to navigate to the GET endpoint.
- Closest apparent control: none; logout is state-changing GET. `SameSite=Lax` permits cookies on a cross-site top-level safe-method navigation, and there is no CSRF token or Origin/Referer verification.
- Sink/broken control: deletion of the presented staff session at 17-22 plus clearing all cookies at 25-28.
- Impact: forced staff logout and loss of unrelated application-origin cookie state; no privilege gain or data disclosure is shown.
- CWE: CWE-352 (Cross-Site Request Forgery).
- Candidate-local validation: from a foreign origin, top-level navigate a logged-in staff browser to `/api/logout`; confirm session deletion and redirect. 
- Attack-path facts / preliminary disposition: public navigation -> ambient Lax cookie -> state-changing GET. **Preliminary: technically valid, low severity / likely suppress from final report unless forced logout is operationally material.**
