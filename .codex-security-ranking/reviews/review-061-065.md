## Stable candidates

### MANSATAS-061-SETTINGS-UNAUTH-YEAR-MUTATION — unauthenticated server actions can rewrite or delete academic-year state

- Severity: **Critical**.
- Instance key: `missing-authz:app/dashboard/settings/actions.ts:18`.
- Affected locations: `app/dashboard/settings/actions.ts:18-57` (create); `118-211` (activate and restore class assignments); `217-237` (delete); `243-263` (jurusan); `269-317` (lesson-time configuration). The client imports and invokes these exports at `app/dashboard/settings/components/settings-client.tsx:22-23,530,574,584,607,619`.
- Source: unauthenticated browser-controlled server-action arguments: `FormData`, academic-year IDs, client-supplied `isActive`, jurusan arrays, and lesson schedule structures.
- Closest apparent control: none inside these exported `'use server'` actions. The settings page separately authenticates and calls `checkFeatureAccess(..., 'settings')` at `app/dashboard/settings/page.tsx:40-46`, but a prior page render/redirect is not an action-boundary control. In particular, `hapusTahunAjaran` trusts the caller's `isActive` boolean at 217-220 rather than reading the row's state before deletion.
- Sink / broken control: inserts to `tahun_ajaran` at 42; class-history snapshot/restore and global active-year updates at 135-189; `dbDelete('tahun_ajaran')` at 221-224; and configuration updates at 247 and 294-296.
- Impact: an internet caller who can invoke the exported action can create junk periods, activate an arbitrary period, reassign every active student's current class from historical snapshots, or attempt deletion of even the active period by passing `false`. The schema uses `ON DELETE CASCADE` from `penugasan_mengajar`, attendance summaries, and `riwayat_kelas` to `tahun_ajaran` (`migrations/schema.sql:156-169,189-193`), so a successful deletion can systemically destroy academic relationships/history. This also corrupts the active-year trust boundary used throughout the application.
- CWE: CWE-306 (Missing Authentication for Critical Function), CWE-862 (Missing Authorization), CWE-602 (Client-Side Enforcement of Server-Side Security; client `isActive`).
- Candidate-local validation: in a disposable local database, capture the action identifiers from a build, call each action without cookies, and verify mutation. For the least destructive proof, invoke `tambahTahunAjaran`; for activation, seed one historical `riwayat_kelas` row and observe `siswa.kelas_id` plus `tahun_ajaran.is_active` change. Test deletion only inside a disposable transaction/database and confirm cascaded row counts. Proof gap: Next production action identifiers/protocol are build-specific, but the exports are intentionally used by a client component and no server-side identity check exists once dispatch reaches them.
- Attack-path facts: unauthenticated browser -> Next server-action dispatch -> exported function -> privileged D1 binding -> global academic-year/class mutation. No session, feature, role, object, or operation check intervenes. **Preliminary disposition: reportable, high confidence.**

### MANSATAS-061-DASHBOARD-CONFIG-AUTHZ — any authenticated account can change global dashboard visibility and widget configuration

- Severity: **Medium**.
- Instance key: `operation-authz:app/dashboard/settings/actions.ts:63`.
- Affected locations: `app/dashboard/settings/actions.ts:63-85` and `91-112`; expected page control at `app/dashboard/settings/page.tsx:40-46`.
- Source: arbitrary nested visibility/widget maps submitted by any signed-in staff account through direct server-action invocation.
- Closest apparent control: `getCurrentUser()` rejects anonymous callers at 64-65 and 92-93. There is no `settings` feature check or administrative role check. Input cleaning only constrains the data shape/catalog and does not authorize the operation (67-76, 95-103).
- Sink / broken control: global `setSystemSetting` calls at 79 and 106, followed by dashboard revalidation.
- Impact: a low-privileged account can globally hide dashboard entries by group/role or alter extra widget composition, interfering with availability and administrative policy for other users. It does not directly grant the attacker a hidden feature because downstream feature checks may still apply.
- CWE: CWE-862 (Missing Authorization), CWE-863 (Incorrect Authorization).
- Candidate-local validation: sign in as a role without `settings`, invoke both actions directly, then read the two global settings or reload affected dashboards and observe the change. Proof gap: exact consumer interpretation of attacker-chosen group keys should be verified, although durable unauthorized global writes are statically established.
- Attack-path facts: authenticated low-privilege user -> direct action -> session-only check -> global settings write. **Preliminary disposition: reportable integrity/availability issue.**

### MANSATAS-061-STUDENT-ACTIVE-YEAR-SCOPE — historical teacher and BK mappings grant current student list/detail/export access

- Severity: **High** because the stale scope reaches a bulk export of national IDs, family identity and contact data; the ordinary list/detail instances alone are medium.
- Instance key: `academic-year-object-authz:app/dashboard/siswa/actions.ts:381`.
- Affected locations: `app/dashboard/siswa/page.tsx:91-106` (class-scope derivation) and `34-61` (list sink); `app/dashboard/siswa/actions.ts:379-415` (export); `app/dashboard/siswa/[id]/page.tsx:68-75` (BK detail authorization). Supporting schema/control evidence: `penugasan_mengajar.tahun_ajaran_id` at `migrations/schema.sql:156-162`; `kelas_binaan_bk.tahun_ajaran_id` and period-unique mapping at `migrations/schema-migration-bk-ta.sql:7-13`; correct negative controls bind the requested period in `app/dashboard/bk/actions.ts:95-101,136-143,204-209`.
- Source: an authenticated teacher or guru BK with a mapping from any historical year, plus an attacker-selected student ID for detail or a direct bulk-export invocation.
- Closest apparent control: the list page checks authentication and `siswa` feature access (79-84); export checks only authentication (367-368); detail checks authentication and BK role/current-class ID (23-24, 68-70). However, teacher queries constrain only `guru_id`, and BK queries constrain only `guru_bk_id`; none joins/binds the row where `tahun_ajaran.is_active = 1` (list 102-106; export 381-385; detail 68-70).
- Sink / broken control: historical class IDs are used to fetch current occupants from mutable `siswa.kelas_id` on the list (34-61), to release complete export rows (393-415), and to authorize detailed records (32-48, 94-184, 266-280). Because class IDs are stable and reused while active student placement changes, an old assignment silently confers access to today's occupants.
- Impact: a teacher/BK whose relationship ended in a prior year can enumerate current students of the reused class. Export escalates this to bulk disclosure of NISN, NIK, KK number, address, WhatsApp, parent NIK, parent birth data and income for all such current occupants. BK can additionally open the detailed discipline, permissions, grades, and finance view.
- CWE: CWE-639 (Authorization Bypass Through User-Controlled Key / stale object relationship), CWE-862 (Missing Authorization), CWE-200 (Exposure of Sensitive Information).
- Candidate-local validation: seed active year B and an old year-A `penugasan_mengajar` or `kelas_binaan_bk` row for user U/class K, with no year-B row; place a current student in K. As U, load `/dashboard/siswa`, invoke `getSiswaExportData`, and for BK open `/dashboard/siswa/{id}`. Expect list/export/detail access despite no active assignment. Then add `tahun_ajaran_id = active.id` to the candidate query in a test branch and verify denial. No deployment-state proof is required beyond at least one retained historical mapping, which the year-scoped schema is designed to preserve.
- Attack-path facts: authenticated former assignee -> historical relationship row -> unfiltered class-ID set -> current `siswa.kelas_id` -> PII/detail/export. **Preliminary disposition: reportable, high confidence.**

### MANSATAS-061-STUDENT-BULK-EXPORT-AUTHZ — direct export bypasses the feature gate and grants broad roles all sensitive student records

- Severity: **High**.
- Instance key: `bulk-sensitive-export:app/dashboard/siswa/actions.ts:366`.
- Affected locations: `app/dashboard/siswa/actions.ts:366-415`; comparison page control at `app/dashboard/siswa/page.tsx:79-96`.
- Source: direct invocation by any authenticated account. Role resolution at 371 is used only to decide full-list versus related-class scope.
- Closest apparent control: authentication at 367-368 and role lookup at 371. Unlike the page, the action never calls `checkFeatureAccess(..., 'siswa')`. Roles `resepsionis`, `guru_piket`, and `guru_ppl` are explicitly classified for full-list access at 372-374, along with leadership/TU. No independent export permission or sensitive-field policy is checked.
- Sink / broken control: the full-list branch leaves `where` empty (376-394), and the SELECT returns every status, including soft-deleted/departed rows, with NISN/NIK/KK/address/contact/parent NIK/birth/income columns at 396-413; the complete array crosses the server boundary at 415.
- Impact: any account with one of the enumerated operational roles can export the entire database's sensitive student and parent identity data, even when its `siswa` feature is disabled and even though ordinary page access would redirect. A generic authenticated role without those role strings still receives sensitive rows for any stale related-class mapping (tracked separately above).
- CWE: CWE-862 (Missing Authorization), CWE-200 (Exposure of Sensitive Information), CWE-359 (Exposure of Private Personal Information to an Unauthorized Actor).
- Candidate-local validation: create a `guru_piket`/`resepsionis` user with no `siswa` role-feature grant, verify `/dashboard/siswa` redirects, then directly invoke `getSiswaExportData` and confirm rows contain `nik`, `nomor_kk`, `nik_ayah`, `nik_ibu`, addresses, contacts, and incomes. Also seed a `status='dihapus'` row and confirm it is exported. Proof gap: whether those roles are assigned in production affects immediate population, not the reachable authorization defect.
- Attack-path facts: authenticated operational account -> direct action (page feature gate bypassed) -> broad role string -> unfiltered full-table query -> bulk PII response. **Preliminary disposition: reportable, high confidence.**

### MANSATAS-061-STUDENT-DETAIL-OVEREXPOSURE — wali/BK detail authorization releases identity, behavioral and financial records without feature- or data-domain checks

- Severity: **Medium** (bounded to a class, but the fields are highly sensitive; combine with the stale-BK candidate where historical scope exists).
- Instance key: `sensitive-field-authz:app/dashboard/siswa/[id]/page.tsx:32`.
- Affected locations: `app/dashboard/siswa/[id]/page.tsx:23-75` (entry and object control), `32-48` (identity/household source), `94-184` (behavioral, academic and finance sources), and `266-280` (client serialization sink). UI confirmation: `app/dashboard/siswa/[id]/components/detail-client.tsx:75-77,343,351-403,609-743` restricts editing to admin/TU but visibly exposes biodata and the finance tab to every authorized viewer.
- Source: attacker-selected route `id` from an authenticated `wali_kelas` or `guru_bk` account.
- Closest apparent control: the page allows admin/leadership, or checks only whether the caller is wali/BK for the student's class (59-75). It does not call the `siswa` feature gate used by the list page, and performs no separate finance, discipline, or sensitive-biodata permission check. Editing is restricted in the client, but read authorization is not.
- Sink / broken control: full student/parent NIK, KK, home address, phone and income values plus discipline, permission, grades, DSPT/SPP balances and transaction receipt data are passed to a client component. Selecting another tab is not required because all promises execute and all props are serialized on initial render.
- Impact: a wali or BK account can read identity-theft-grade family data, student behavioral records, and financial/payment history for every student in its allowed class even when feature management has disabled `siswa` access and without a finance role. This is a bounded class-size disclosure rather than a school-wide export.
- CWE: CWE-862 (Missing Authorization), CWE-200/CWE-359 (Sensitive personal information exposure).
- Candidate-local validation: revoke the `siswa` feature from a wali/BK test user while preserving its class relationship, request a known related `/dashboard/siswa/{id}`, and inspect the rendered/RSC payload for NIK/KK/address and finance objects. Separately decide with the product owner whether wali/BK are intentionally entitled to each domain; no code evidence shows such a policy gate.
- Attack-path facts: role plus class relationship -> direct detail route bypassing feature management -> unconditional multi-domain queries -> serialized sensitive props. **Preliminary disposition: reportable unless an explicit institutional policy grants wali/BK full finance and national-ID access.**
