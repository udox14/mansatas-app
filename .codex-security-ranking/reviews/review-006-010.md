## Raw candidates (pre-dedupe)

### R006-010-WA-001 — Wablas webhook POSTs are accepted without authentication

- Suggested severity: **Medium**.
- CWE: **CWE-306, Missing Authentication for Critical Function**.
- Source: unauthenticated Internet POST body and its `id`/`message_id`/`messageId`, `status`/`ack`, and error/message properties, `app/api/whatsapp/webhook/route.ts:21-22,44-53`; the same payload can also enter the Meta-shaped loop at `:57-64`.
- Closest control: `app/api/whatsapp/webhook/route.ts:25-35` appears to require a valid provider signature. The called implementation is `lib/whatsapp.ts:672-675`, where `provider === 'wablas'` returns `true` unconditionally. Wablas is also the default provider (`lib/whatsapp.ts:8,46`). Although `wablasSecretKey` is loaded (`lib/whatsapp.ts:54,67`), it is not used by this webhook verifier.
- Sink: `updateWhatsAppMessageStatus` calls at `app/api/whatsapp/webhook/route.ts:53,64`; concretely, `lib/whatsapp.ts:656-669` updates `wa_outbox.status`, `wa_outbox.error_message`, and the related `wa_campaign_recipients.status` by attacker-supplied provider message ID. Unknown status values are coerced to `sent` at `lib/whatsapp.ts:658-660`.
- Impact: an unauthenticated caller who has or learns a provider message ID can falsify sent/delivered/read/failed state and error details for school WhatsApp notifications, corrupting delivery/audit state and campaign-recipient workflow integrity. It does not send a new message or alter recipient/student records through this sink.
- Candidate-local validation:
  - Confirmed control bypass: for the default/configured `wablas` mode, no header or secret is required; the route's `signatureValid` becomes true at line 31 via the unconditional helper return.
  - Confirmed reachable mutation: a body such as `{"id":"<known-provider-id>","status":"read"}` reaches line 53 after JSON parsing and updates matching rows through parameterized SQL.
  - Binding protects against SQL injection but not authorization; `providerMessageId`, `status`, and `errorMessage` are attacker-selected data bound to an unauthenticated operation.
  - Exploit precondition: the attacker needs a valid `provider_message_id` for a meaningful existing-row update. Repository searches did not find that field rendered by application pages, so acquisition is not proven locally; IDs may still be known to recipients, integrations, logs, or provider-side actors. This limits blast radius/severity but does not restore authentication.
- Candidate-local attack path:
  1. Internet attacker sends POST to `/api/whatsapp/webhook`; the threat model explicitly treats webhooks as an unauthenticated boundary requiring strong secrets and fail-closed behavior.
  2. Default Wablas provider causes `verifyWhatsAppSignature` to return true without examining any signature.
  3. Route parses attacker JSON and derives an arbitrary provider message ID/status/error.
  4. `updateWhatsAppMessageStatus` modifies matching outbox and campaign-recipient rows.
  5. Staff-facing WhatsApp state is false. No subsequent provenance check, monotonic transition check, or ownership check is present.
- Disposition: **reportable candidate; forward to validation and attack-path analysis**. Keep separate from R006-010-WA-002 because it is an always-open default-provider branch rather than a deployment-secret omission.

### R006-010-WA-002 — Meta-compatible webhook verification fails open when the app secret is absent

- Suggested severity: **Medium**, configuration-dependent.
- CWE: **CWE-636, Not Failing Securely ('Failing Open')** (also maps to CWE-306).
- Source: unauthenticated POST body and status identifiers at `app/api/whatsapp/webhook/route.ts:21-22,38-64`.
- Closest control: non-Kirimdev providers enter `verifyWhatsAppSignature` through `app/api/whatsapp/webhook/route.ts:29-32`. `lib/whatsapp.ts:673-676` returns `true` when `appSecret` is empty, rather than rejecting the webhook. `getWhatsAppConfig` permits an empty app secret (`lib/whatsapp.ts:43-50`).
- Sink: the same `updateWhatsAppMessageStatus` calls at route lines 53 and 64, implemented by DB updates at `lib/whatsapp.ts:656-669`.
- Impact: in a Meta/cloud-api configuration missing `WHATSAPP_APP_SECRET`/the equivalent system setting, any Internet caller with a valid provider message ID can falsify WhatsApp delivery and campaign-recipient state.
- Candidate-local validation:
  - Confirmed fail-open branch: missing secret returns true before the header format or HMAC is checked.
  - Confirmed mutation and parameter binding are identical to R006-010-WA-001.
  - The repository comments instruct operators to set `WHATSAPP_APP_SECRET` (`wrangler.toml:46`), but neither deployment presence nor runtime value is provable from source. The vulnerable state is nevertheless explicitly accepted by code instead of failing closed.
  - Exploit preconditions: affected deployment must select a non-Wablas, non-Kirimdev provider and omit the app secret; attacker must know a matching provider message ID for an existing-row impact.
- Candidate-local attack path:
  1. Deployment uses Meta/cloud-api behavior but app secret is absent/misconfigured.
  2. Internet attacker POSTs a forged status event without `x-hub-signature-256`.
  3. Helper returns true solely because the secret is empty.
  4. Route writes attacker-chosen state to matching WhatsApp rows.
- Disposition: **reportable/deferred-runtime-proof candidate**. Validation should preserve it as a separate fail-open instance and confirm deployment configuration if available; absence of runtime proof should be stated, not silently treated as safe.
