## Plausible raw candidates

### MANSATAS-001-PARENT-DEFAULT-CREDENTIAL — parent authentication remains bypassable with public/guessable identity data

- Instance key: `auth-default-credential:app/api/auth/[...all]/route.ts:52`
- Affected locations: `app/api/auth/[...all]/route.ts:52-60` (entrypoint/wrapper); `utils/auth/index.ts:400-420` (concrete implementation / account lookup); `utils/auth/index.ts:436-468` (root control and credential acceptance); `utils/auth/index.ts:504-535` (session-creation sink).
- Source: unauthenticated `nisn` and `password` JSON fields. An attacker can submit a student's NISN as both identifier and password, or use the student's `DDMMYYYY` birth date.
- Closest apparent control: a PBKDF2 credential is stored in `parent_credentials` and checked at 440-441. It is bypassed by the unconditional fallback at 442-458, which accepts NISN/birth date even after a custom hash exists and then overwrites that hash. The no-credential path explicitly accepts the same defaults at 459-468. Format regexes at 403-410 do not establish secret entropy.
- Sink/broken control: issuance of a 30-day parent session for `siswa.id` at 504-535.
- Impact: takeover of a parent's portal identity and access to that student's parent-visible PII, finance/payment, and academic records; an attacker can also reset the stored parent password merely by authenticating with the fallback.
- CWE: CWE-521 (Weak Password Requirements), CWE-798 (Use of Hard-coded Credentials; predictable per-record default), CWE-287 (Improper Authentication).
- Candidate-local validation: local code proves the fallback remains enabled after password enrollment. Validate with a test student whose password has been changed: POST the NISN or birth date and confirm a session plus replacement hash. Proof gap: external availability/guessability of a target NISN/DOB and the exact parent portal data reachable are not established in this slice.
- Attack-path facts / preliminary disposition: internet client -> public parent sign-in branch -> normalized student lookup -> fallback comparison -> durable session. No role, prior-session, one-time-enrollment, or forced-reset requirement intervenes. **Preliminary: reportable; high-confidence auth bypass, severity at least medium and potentially high by portal blast radius.**

### MANSATAS-001-PARENT-BRUTEFORCE — public parent login has no attempt throttling

- Instance key: `auth-bruteforce:app/api/auth/[...all]/route.ts:52`
- Affected locations: `app/api/auth/[...all]/route.ts:52-60` (entrypoint); `utils/auth/index.ts:400-471` (credential validation/root control).
- Source: unlimited unauthenticated requests containing NISN and 6-20 digit password candidates.
- Closest apparent control: input is digit-normalized and format-limited at `utils/auth/index.ts:400-410`; PBKDF2 is used only when a stored hash is tried at 440-441. There is no IP/account/session throttle, lockout, progressive delay, CAPTCHA, or attempt ledger in the route or implementation.
- Sink/broken control: repeated password oracle ending in session issuance at 504-535; repeated PBKDF2 work also consumes Worker CPU.
- Impact: practical online guessing is materially amplified by the NISN/DOB fallback; possible targeted account takeover and resource exhaustion.
- CWE: CWE-307 (Improper Restriction of Excessive Authentication Attempts).
- Candidate-local validation: send repeated failed attempts and observe that every request reaches lookup/password work and none returns 429/lockout. Proof gap: an upstream Cloudflare WAF/rate-limit rule outside the repository may mitigate deployment.
- Attack-path facts / preliminary disposition: fully public route, deterministic credential oracle, low-entropy accepted credential family. **Preliminary: validation recommended; report separately from the predictable-default root cause because upstream rate limiting could close only this row.**

### MANSATAS-001-STAFF-BRUTEFORCE — both staff login entrypoints lack attempt throttling

- Instance key: `auth-bruteforce:app/(auth)/login/actions.ts:9`
- Affected locations: `app/(auth)/login/actions.ts:9-25` and `app/api/auth/[...all]/route.ts:17-26` (independent entrypoints); `utils/auth/index.ts:238-278` (shared implementation/root control/session sink).
- Source: unauthenticated email/password in FormData or JSON.
- Closest apparent control: PBKDF2 verification occurs at `utils/auth/index.ts:264`, but neither caller nor the shared implementation records failures, delays attempts, locks an account, or returns 429. Banned-user rejection at 251-254 is not an attempt control.
- Sink/broken control: unlimited credential oracle and successful 30-day staff session insertion at 270-278.
- Impact: password spraying/credential stuffing against staff and privileged accounts, plus avoidable Worker CPU/database load.
- CWE: CWE-307.
- Candidate-local validation: execute a bounded request series against both entrypoints and verify no throttle or lockout. Proof gap: password policy and external Cloudflare rate-limiting configuration are outside this review.
- Attack-path facts / preliminary disposition: two public paths reach the same verifier, so rate-limiting only one URL would remain bypassable. **Preliminary: plausible, validation recommended; severity depends on external controls and password policy.**

### MANSATAS-001-STAFF-TOKEN-JSON — HttpOnly staff token is returned in readable JSON

- Instance key: `session-secret-exposure:app/api/auth/[...all]/route.ts:74`
- Affected locations: `app/api/auth/[...all]/route.ts:73-79` (entrypoint/serialization sink); `utils/auth/index.ts:368-392` (root control / concrete returned object).
- Source: any browser request carrying a valid session cookie, including one made by same-origin injected script.
- Closest apparent control: the cookie is `HttpOnly` (`utils/auth/index.ts:59-62` and login action 29-35), but `getSession` returns `row.token` at 389-392 and the route serializes the entire object at 78, undoing the confidentiality property. Authentication proves possession but does not justify disclosing the reusable bearer secret to script.
- Sink/broken control: `NextResponse.json(session)` exposes the 30-day bearer token in the response body.
- Impact: same-origin XSS can exfiltrate a durable staff session token and replay it outside the victim browser, turning transient script execution into account takeover.
- CWE: CWE-200 (Exposure of Sensitive Information), CWE-922 (Insecure Storage of Sensitive Information) / session-secret exposure.
- Candidate-local validation: authenticated GET should show `session.token` equal to the cookie value. Proof gap: a same-origin script execution primitive is needed for remote exfiltration; candidate `MANSATAS-001-R2-STORED-XSS` supplies one plausible primitive.
- Attack-path facts / preliminary disposition: authenticated cookie -> JSON endpoint -> raw bearer token -> attacker-controlled network request. **Preliminary: reportable defense/control break, likely medium when chained with XSS.**

### MANSATAS-001-R2-UNAUTH-READ — unauthenticated arbitrary-key R2 proxy exposes private school files

- Instance key: `object-authz:app/api/media/[...path]/route.ts:13`
- Affected locations: `app/api/media/[...path]/route.ts:8-25` (public entrypoint and broken root control); `app/api/media/[...path]/route.ts:31-54` (read/response sink). Supporting concrete sensitive writers/key schemes: `utils/r2.ts:100-142` (student photos, user avatars/signatures, private-cache S36 PDFs), `utils/r2.ts:148-183` (discipline evidence, payment proofs, attendance photos), and `app/api/pmb/download-berkas/route.ts:9-19,41-49` (negative control showing PMB identity files normally require privileged roles).
- Source: unauthenticated path segments; `path.join('/')` directly chooses any bucket key.
- Closest apparent control: only non-empty key and object-exists checks at 20-29. There is no session, role, family/student ownership, prefix allowlist, signed URL, per-object ACL, or database authorization lookup. The repository has no `middleware.ts`/`proxy.ts` global guard. R2 itself returns any matching key to the privileged Worker binding.
- Sink/broken control: `env.R2.get(key)` and inline response of full object bytes.
- Impact: disclosure of student photos, signatures, performance documents, discipline/attendance evidence, payment proofs, and potentially admissions identity documents when an attacker knows or predicts a key. Public one-year immutable caching (18, 50-53) can prolong disclosure after authorization or underlying content changes.
- CWE: CWE-862 (Missing Authorization), CWE-639 (Authorization Bypass Through User-Controlled Key), CWE-200.
- Candidate-local validation: unauthenticated GET a known `tpg/s36/{userId}.pdf`, `signatures/{userId}/signature.png`, `foto_siswa/{siswaId}/foto.jpg`, or `presensi/{date}/{userId}_{action}.jpg` and verify 200. Proof gap: obtain representative object keys from seeded/test data and verify whether PMB uploads share this binding/prefix.
- Attack-path facts / preliminary disposition: internet path -> arbitrary key -> privileged R2 binding -> bytes, with no intervening identity check. IDs/dates and URLs are reused in application data/UI, while several key schemes are deterministic. **Preliminary: reportable high-confidence broken object authorization; likely high due breadth and sensitive documents.**

### MANSATAS-001-R2-STORED-XSS — attacker-selected R2 MIME type is served inline on the application origin

- Instance key: `stored-xss:app/api/media/[...path]/route.ts:31`
- Affected locations: `app/dashboard/buku-tamu/actions.ts:35-60,72-103` (supporting authenticated input/writer); `app/api/media/[...path]/route.ts:31,45-54` (root control and response sink).
- Source: authenticated user's `foto_base64` bytes and unrestricted `foto_content_type` supplied to `simpanEntriTamu`; the writer stores that value as R2 `httpMetadata.contentType` without validating bytes or MIME.
- Closest apparent control: the supporting action requires any current user at 80-81 and coerces only the filename extension to `.jpg`/`.webp` at 52-53. It does not restrict `contentType`; the media route trusts stored metadata at 31, provides neither `Content-Disposition: attachment` nor `X-Content-Type-Options: nosniff`, and serves bytes inline.
- Sink/broken control: an object containing HTML/script can be returned as `text/html` from `/api/media/buku-tamu/{known-id}.jpg` on the main application origin.
- Impact: persistent same-origin script execution when a victim visits/opens the object; script can call the two get-session endpoints and exfiltrate raw staff/parent bearer tokens, access same-origin data, or perform privileged actions in the victim session.
- CWE: CWE-79 (Improper Neutralization of Input During Web Page Generation), CWE-434 (Unrestricted Upload of File with Dangerous Type).
- Candidate-local validation: as a low-privileged user, submit base64 HTML with `foto_content_type: text/html`, use the returned `id` to request `/api/media/buku-tamu/{id}.jpg`, and verify script execution plus response headers. Proof gap: confirm ordinary low-privileged roles can invoke the server action directly in deployed Next.js action routing and whether a CSP blocks inline/script network exfiltration.
- Attack-path facts / preliminary disposition: authenticated low-privilege input -> R2 metadata/body -> unauthenticated same-origin media route -> privileged victim navigation -> session JSON/token exfiltration. **Preliminary: strong chained candidate; validation recommended, potentially high.**
